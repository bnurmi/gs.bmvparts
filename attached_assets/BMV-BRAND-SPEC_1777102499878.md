# BMV.parts — Brand & UX Direction

This document is the source of truth for the BMV.parts visual system. Implement against it. When the spec and the existing code disagree, the spec wins.

---

## The one-paragraph brief

BMV.parts is a **reference tool**, not a shopfront. It looks like an instrument workshops open before they open ETK itself: dense, calm, fast, precise. The visual system is **paper-and-ink with one decisive blue**, monospace for every part number and code, and zero decoration. The three-panel drill-down stays exactly as it is — what changes is the typographic register, the use of colour, and the density of information per surface.

---

## Foundational rules (apply everywhere)

1. **Paper and ink first; blue only for actions.** If the user can't click it, or it isn't the brand mark, it isn't blue. Surfaces default to paper or ink. No saturated blue cards, no decorative blue fills.
2. **Monospace is the secret sauce.** Every part number, VIN, chassis code, hex, file size, count, and timestamp is `JetBrains Mono`. This is what makes the product feel engineered.
3. **One primary action per page.** Hero CTA on homepage = VIN input. Primary on car detail = the diagram. Primary on part detail = the part number + price. Everything else is secondary.
4. **Rulers, not shadows.** Structure comes from 1px rules between paper-tone surfaces. Shadows are reserved for the single hero CTA (hard offset) and floating menus (soft). No decorative shadows.
5. **Density is a feature.** Workshops want dense reference data. Tighten, don't thin. Don't pad rows. Don't collapse desktop tables into mobile cards.
6. **Label everything quietly.** Every section, column, and stat carries a small monospace label in BMV Blue at 10–11px, uppercase, with `letter-spacing: 0.14em`. These micro-labels are what transform UI into documentation.

## Tokens

Use the tokens from `bmv-tokens.css` and `tailwind.config.js`. Don't introduce new colours, fonts, radii, or shadows outside this system. If something doesn't fit a token, the design — not the token — is wrong.

### Colour

| Token | Light value | Dark value | Use |
|---|---|---|---|
| `--surface-base` | `#F5F6F9` | `#0C0F14` | Page background |
| `--surface-raised` | `#FFFFFF` | `#141821` | Cards, content blocks, inputs |
| `--surface-sunken` | `#ECEEF2` | `#181C26` | Table headers, nested fills |
| `--surface-sidebar` | `#08090B` | `#08090B` | Sidebar — stays dark in both themes |
| `--fg-primary` | `#10131A` | `#F0F2F6` | Body, headings |
| `--fg-tertiary` | `#6B7685` | `#8892A0` | Metadata, helper text |
| `--accent` | `#1563D6` | `#4F94F0` | Links, active state, brand mark |
| `--accent-fog` | `#E5EEFB` | `rgba(79,148,240,0.10)` | Active item background |
| `--state-signal` | `#C2881A` | `#E8B84A` | Prices, supersession, warnings |
| `--state-success` | `#2F8A5B` | `#4FB07A` | Sync chips, "Complete" status |
| `--border-default` | `#E6E8ED` | `#222834` | Default rules between surfaces |
| `--border-ink` | `#10131A` | `#F0F2F6` | Hero CTA border, primary buttons |

**Rule:** the only place pure white appears in light mode is on `--surface-raised`. The only place anything close to pure black appears is `--fg-primary` and `--surface-sidebar`. Everything else is on the warm-grey scale.

### Typography

Two families, no exceptions:

- **Inter Tight** — every UI surface, every heading, every body paragraph. Weights 300, 400, 500, 600, 700, 800, 900.
- **JetBrains Mono** — every part number, VIN, count, code, hex, timestamp. Weights 400, 500, 600.

**No serifs. No italics. No script faces.** Emphasis inside headings is done by dropping to Inter Tight 300 and colouring it `--accent`. Same family, weight contrast does the work.

### Radius

Maximum corner radius is **2px**. Anything more starts looking like a default UI kit. Most components use no radius at all.

### Shadows

There are exactly **two** shadows in the system:

- `shadow-hero` — `4px 4px 0 var(--border-ink)`, hard offset, no blur. Used on the single hero CTA on homepage. On hover, becomes `5px 5px 0`.
- `shadow-floating` — soft, low-opacity. Used only for popover menus, dropdowns, modals.

If a designer or PR adds a third shadow, reject it.

## Components

### Sidebar (always dark)

The sidebar is `--surface-sidebar` in both light and dark mode. This is intentional: it visually anchors navigation away from content and lets the dense monospace part numbers in the main area sing. Inside the sidebar:

- Active nav item has a 2px `--accent` left border, `--surface-sidebar-raised` background, and white text
- Inactive items use `--fg-tertiary` (steel) and white-on-hover
- The "M Models" list shows the **top 8** by parts count with a "Show all 23 →" affordance — never the full list by default
- Counts are monospace, `--fg-quiet`, right-aligned
- "AI" badges on Part Finder use the accent fill with white text

### Topbar

After removing the duplicate logo, the topbar becomes: **universal search input** (taking ~540px max-width, with ⌘K hint) + a status chip (e.g. "Catalog live · 24h sync" in success colour) + sign-in button. Border-bottom 1px `--border-default`.

### Hero CTA (homepage only)

The single primary action: a bordered input that takes a VIN, part number, or chassis code, with:
- Border `--border-ink` (1px solid)
- Hard offset shadow `shadow-hero`
- Monospace placeholder text in `--fg-quiet`
- A small monospace label on the left ("VIN / PART / CHASSIS") in `--fg-quiet` with `letter-spacing: 0.14em`
- A solid ink button on the right with white text and "Decode →"
- On hover: shifts up 1px, shadow extends to `shadow-hero-hover`

This is the **only** element on the homepage that uses a hard-offset shadow. Don't replicate the treatment on other CTAs.

### Car cards (homepage browse grid)

Replace the current saturated-blue cards with paper containers:
- White (or `--surface-raised`) background, 1px `--border-default` rule (use a 1px gap on the grid, no per-card border)
- Inside each card:
  - Chassis code in monospace 10px BMV Blue uppercase ("E82 · 2011–2012")
  - Model name in Inter Tight 600 17px (e.g. "E82 M Coupé")
  - Engine / body in Inter Tight monospace 12px `--fg-quiet`
  - 84×52 image thumb on the right with a 1px `--border-default` rule
  - Three monospace stat cells (Cats / Groups / Parts) divided by 1px rules
  - Footer: "Complete · DD·MM·YY" sync status (success colour) + "Browse →" link
- Hover: background shifts to `--surface-quiet`. No transform, no shadow.

### Three-panel part detail (KEEP GEOMETRY EXACTLY)

The three-column drill-down (Categories → Subcategories → Diagram + Parts) stays. Only typographic and colour treatment changes:

**Column 1 — Categories**
- Each item has a small `▸` caret (collapsed) or `▾` (active) in `--fg-quiet`
- Active category gets `--surface-quiet` background + 2px `--accent` left border
- 13px Inter Tight, weight 500 when active

**Column 2 — Subcategories**
- Each item has a 22×22px icon thumbnail (light grey square in unselected state)
- Active subcategory uses `--accent-fog` background, 2px `--accent` left border, `--accent-hover` text colour
- Item count on the right in monospace 10.5px

**Column 3 — Diagram + Parts**
- Header band: monospace breadcrumb + Inter Tight 20px title on left, monospace ref number on right
- Diagram area on `--surface-base` background, min 260px tall, with a 1px `--border-default` separator above the parts table
- Parts table:
  - Header row in `--surface-base`, monospace 10px uppercase column titles in `--fg-quiet`
  - Each row has: index (monospace), part number (mono in BMV Blue) with raw form below in `--fg-quiet`, description with mono spec line below, qty in mono, and **EU dealer price in its own column** (mono, with AUD approximation in smaller `--fg-quiet` text)
  - Row hover: background shifts to `--surface-base`
  - 1px `--border-default` between rows

**Critically:** active selection in light mode uses `--accent-fog` (a 10%-tinted blue), not solid blue fill. The current site uses 100% saturated blue for the active sub-item — replace with the fog treatment everywhere.

### Tables generally

Every table on the site (parts, click logs, backups, users, API keys) follows the same pattern:
- Monospace column headers, uppercase, 10px, letter-spacing 0.14em, `--fg-quiet`
- Monospace data cells for IDs, numbers, codes, timestamps
- Inter Tight for description fields
- 1px `--border-default` between rows
- No zebra striping. Hover state is `--surface-base`.

### Buttons

Three button styles only:

1. **Primary** — `--border-ink` background, `--fg-inverse` text, no radius, 14px Inter Tight 500. Hover lightens to `--accent`.
2. **Secondary** — Transparent background, 1px `--border-ink` border, `--fg-primary` text. Same dimensions.
3. **Ghost link** — No background, `--fg-primary` text with a 1px `--border-default` underline that becomes `--accent` on hover.

No tertiary buttons, no "destructive" red buttons (use the modal confirmation page approach for destructive actions, like the existing backup restore flow).

### Status chips

Inline status indicators (e.g. "Catalog live · 24h sync", "Fully synced · 08·03·26"):
- Monospace 10.5px, uppercase, letter-spacing 0.08em
- 6px coloured dot prefix
- Colour from state tokens: `--state-success`, `--state-signal`, `--state-error`

### Forms

- Input fields: 1px `--border-default`, `--surface-raised` background (or `--surface-base` for "filter" inputs), 8–14px padding
- Focus state: border becomes `--accent`, no glow, no ring
- Labels: Inter Tight 500 13px, paired ABOVE the input (not floating, not inline)
- Helper text: 12px `--fg-tertiary` below the input
- Error text: 12px `--state-error` below the input

## Dark mode

Dark mode is a **re-skin**, not a redesign. Same hierarchy, same density, same components, same primary actions.

### How dark mode works

A theme toggle in the topbar (or in the user dropdown) cycles through `light` → `dark` → `auto`. The choice persists in `localStorage` under `bmv-theme`. The `auto` mode follows `prefers-color-scheme`. Use `theme-toggle.js` from the package — drop it inline into the HTML `<head>` BEFORE any CSS to prevent flash-of-unstyled-content.

The mode is applied via `data-theme="..."` on `<html>`, not a class. Tailwind config is set up to read this attribute.

### Key palette shifts

- Backgrounds invert from paper-grey to a warm near-black scale (`#0C0F14` → `#181C26`). Never pure black.
- Foreground softens from `#10131A` to `#F0F2F6` — never pure white. Reduces eye strain on long catalog browsing sessions.
- BMV Blue brightens from `#1563D6` to `#4F94F0` — the original hue lacks luminance on dark surfaces.
- The sidebar is **already dark** in light mode; it stays dark in dark mode but its border becomes more visible against the new (also-dark) main content.
- The hero CTA's hard-offset shadow changes colour: `--border-ink` flips from near-black (light mode) to soft-white (dark mode), so the shadow stays visible against the dark backdrop.
- Active state fog goes from a 10%-tinted solid blue to a 10%-opacity rgba — same visual weight, different mechanic.

### Things that DON'T change in dark mode

- The three-panel layout, column widths, density, and spacing
- The logo construction (just swap `bmv-logo-header.svg` for `bmv-logo-header-dark.svg`)
- Monospace usage and small labels
- The single primary CTA per page

### Logo swapping

In React, prefer `<img src={logoUrl}>` where `logoUrl` reads from a CSS variable, OR conditional render based on the resolved theme. The CSS-variable approach is simpler and avoids a hydration mismatch:

```css
.app-logo {
  background-image: var(--logo-url);
  background-repeat: no-repeat;
  background-size: contain;
}
```

### Charts and analytics

The Analytics tab will need bespoke dark-mode treatment:
- Bar / line chart fills use `--accent` and `--state-signal` at full saturation in light mode
- In dark mode, these brighten to `--brand-blue-bright` and `--state-signal-bright`
- Gridlines use `--border-default` in both modes (it's already theme-aware)
- Axis labels use `--fg-tertiary`

## Migration notes (for the agent)

- **Don't rewrite layouts.** The three-panel drill-down, the sidebar with model counts, the dashboard metric tiles — all of these have correct *structure*. The work is restyling.
- **Replace shadcn/default-kit components piecemeal.** Buttons, inputs, badges, table rows. Don't rebuild from scratch.
- **Start with tokens.** Drop `bmv-tokens.css` into `client/src/index.css` as the first import. Drop `tailwind.config.js` in. Once tokens are in place, everything else falls out of refactoring components to use them.
- **Then logo + sidebar.** Replace the duplicate logo, restyle the sidebar to dark, replace the M Models full list with the Top 8 + "Show all" affordance.
- **Then homepage hero.** This is the highest-value visual change.
- **Then car cards.** Restyle the saturated-blue blocks to paper.
- **Then part detail polish + active states.** The blue-fog active treatment in Column 2 is the single most noticeable improvement on Part Detail.
- **Dark mode last.** Once tokens are in place, dark mode is mostly free — drop in `theme-toggle.js`, add a toggle in the user dropdown, verify charts and any hardcoded colours.

If something in the current codebase resists the spec (a component too coupled, a third-party widget that fights the tokens), surface it in a comment rather than working around it.
